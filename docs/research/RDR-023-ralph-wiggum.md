## Summary

Ralph Wiggum is an autonomous development methodology based on continuous AI agent loops. The core concept is a bash loop (`while true; do cat PROMPT.md | claude; done`) that repeatedly feeds an agent a prompt file, allowing iterative work until completion. Each iteration starts with a fresh context window, reads persistent state from disk files, and commits after completing one task. Named after The Simpsons character, it embodies "persistent iteration despite setbacks."

## Key Concepts

| Concept                      | Description                                                                                                                        |
| ---------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| **Fresh Context Per Task**   | Each loop iteration starts with clean context window. The `IMPLEMENTATION_PLAN.md` on disk provides continuity.                    |
| **Three Phases**             | 1) Define Requirements (JTBD → specs) 2) Planning mode (gap analysis → plan) 3) Building mode (implement one task → commit → loop) |
| **Two Prompts**              | `PROMPT_plan.md` for planning (no implementation), `PROMPT_build.md` for building (one task per iteration)                         |
| **Specs as Source of Truth** | `specs/*.md` files define requirements by "topic of concern" (one spec per distinct capability/component)                          |
| **AGENTS.md Operational**    | Brief (~60 lines) file describing how to build/run/test. NOT a changelog. Loaded every iteration.                                  |
| **IMPLEMENTATION_PLAN.md**   | Prioritized bullet list of tasks. Generated by Planning mode. Updated during Building mode. Disposable (regenerate when stale).    |
| **Subagent Fan-Out**         | Use subagents heavily for file reads/searches. Main agent is "scheduler" to preserve context.                                      |
| **Backpressure**             | Tests, typechecks, lints create signals that reject invalid work. Prompt says "run tests"; AGENTS.md specifies actual commands.    |
| **Completion Promise**       | Loop detects `<promise>COMPLETE</promise>` (or similar) to know when to stop.                                                      |
| **Steering via Patterns**    | Add utilities/code patterns to steer Ralph toward correct implementations. "Signs" are anything discoverable.                      |
| **Context = 40-60%**         | Keep context utilization in "smart zone". One task per loop = 100% smart zone utilization.                                         |
| **Dangerously Skip Perms**   | Autonomous mode requires `--dangerously-skip-permissions`. Sandbox becomes only security boundary.                                 |

## Ralph vs AGENTS Comparison

| Aspect                 | Ralph                                           | AGENTS                                          |
| ---------------------- | ----------------------------------------------- | ----------------------------------------------- |
| **Execution Model**    | External bash loop restarts agent fresh         | Single session with mode switching              |
| **Context Management** | Each iteration = clean slate + disk state       | Todo list + handoff files for context anchoring |
| **State Persistence**  | `IMPLEMENTATION_PLAN.md` + `AGENTS.md` on disk  | `.tasks/[NNN]-[slug]/task.md` + phase files     |
| **Phases**             | Planning vs Building prompts                    | Explore → Implement → Review → Commit agents    |
| **Human Checkpoint**   | Before starting; observe/tune from outside loop | After research, after plan, before commit       |
| **Autonomy Level**     | Fully autonomous overnight runs possible        | Human-in-loop at phase transitions              |
| **Target Environment** | Claude Code CLI with bash orchestration         | VS Code Copilot + Claude Code (advisory)        |
| **Tool Restrictions**  | None (sandbox provides safety)                  | Per-agent tool restrictions (Explore read-only) |

## Decision

**Adopted:**

1. **Subagent fan-out for context preservation** — Already adopted in Explore agent (RDR-010). Ralph reinforces: use subagents as "memory extension" to avoid polluting main context.

2. **Disposable plans** — Already adopted. AGENTS' task files serve similar purpose. Reinforces: regenerate plan when trajectory goes wrong rather than fighting it.

3. **Operational docs vs progress** — Ralph's `AGENTS.md` (brief, operational) vs `IMPLEMENTATION_PLAN.md` (status/progress) separation aligns with AGENTS' pattern of keeping agent prompts terse and task files for status.

4. **Backpressure principle** — Tests/lints as "gates" is already implicit in Review agent. Consider explicitly documenting in Review agent: "backpressure via tests rejects invalid work."

**Partially Adopted:**

1. **Fresh context per task** — AGENTS uses handoff between sessions rather than bash loop. The principle applies: avoid context bloat by using task files as external memory.

2. **Spec-driven development** — Similar to [RDR-001](archive/RDR-001-spec-driven.md). AGENTS uses task.md research findings rather than separate spec files. Both achieve "source of truth outside context."

**Rejected:**

1. **External bash orchestration** — AGENTS is designed for interactive IDE use with human checkpoints. Ralph's overnight autonomous mode conflicts with "human-in-loop at leverage points."

2. **Skip permissions** — AGENTS explicitly uses tool restrictions per agent (Explore = read-only). Safety through mode restrictions, not sandbox isolation.

3. **Completion promise pattern** — AGENTS uses explicit handoff buttons/agent mentions rather than magic strings in output.

**Rationale:**

Ralph and AGENTS solve different problems:

- **Ralph**: Maximize autonomous throughput for well-defined tasks (overnight builds, greenfield projects)
- **AGENTS**: Maximize collaboration quality with human oversight at leverage points (real-time development, complex decisions)

Ralph's insights about context management (subagents, disk-based state, disposable plans) are valuable and already partially adopted. But Ralph's core architecture (external loop, skip permissions, autonomous mode) conflicts with AGENTS' philosophy of human-in-loop at research/plan boundaries.

The "40-60% context utilization" and "subagents as memory extension" principles reinforce existing AGENTS patterns without requiring architectural changes.

## Potential Future Adoption

If Claude Code adds a sanctioned "autonomous mode" with proper safety boundaries, consider:

1. **Implement agent loop mode** — Optional mode where Implement agent loops on phases automatically
2. **AGENTS.md operational file** — Add brief operational notes file that persists across sessions (separate from task-specific research)
3. **Acceptance-driven tests** — Ralph's enhancement of deriving test requirements during planning phase could strengthen Review agent

## Comparison to Current Framework

**Overlap:**

- Phase-based workflows (Research/Plan/Implement maps to Planning/Building)
- Context as precious resource (subagent delegation)
- Plan as disposable artifact
- Disk-based state for session continuity

**Conflict:**

- AGENTS: Human reviews research before planning, plan before implementing
- Ralph: Human observes from outside; agent self-corrects through iteration
- AGENTS: Tool restrictions per agent for safety
- Ralph: Sandbox isolation + skip permissions for autonomy
